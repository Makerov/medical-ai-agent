# Story 1.1: Backend Scaffold и Health API

Status: done

## Story

Как разработчик,
я хочу минимальный FastAPI backend scaffold с документированным локальным запуском,
чтобы будущие case, bot, workflow и demo features имели согласованную основу.

## Acceptance Criteria

1. **Дано** свежая копия репозитория, **когда** разработчик запускает backend локально, **тогда** FastAPI app предоставляет health endpoint и generated OpenAPI docs.
2. **И** проект содержит архитектурно заданные директории для API, bots, core, db, models, schemas, services, workflow, workers, integrations, evals, scripts, tests, docs и data.
3. **Дано** backend scaffold существует, **когда** запускаются тесты, **тогда** проходит как минимум один health/API smoke test.
4. **И** configuration загружается через typed settings, а не через hardcoded runtime values.

### Health API Contract

- Recommended endpoint: `GET /api/v1/health`.
- Recommended response shape:

```json
{
  "status": "ok",
  "service": "medical-ai-agent",
  "environment": "local"
}
```

- Keep response typed with a Pydantic schema or explicit response model if it does not introduce unnecessary boilerplate.
- Do not expose stack traces, environment secrets, raw dependency details or host internals through health response.
- This health endpoint is a smoke/readiness signal for local scaffold only. It is not a production liveness/readiness design.

### Configuration Requirements

- `app/core/settings.py` should be the single source for application settings in this story.
- Use typed defaults appropriate for local demo:
  - `APP_NAME=medical-ai-agent`
  - `ENVIRONMENT=local`
  - `API_V1_PREFIX=/api/v1`
  - `DEBUG=false`
  - `LOG_LEVEL=INFO`
- `.env.example` documents expected variables and contains no real tokens, Telegram IDs, database passwords for production, or patient data.
- If using `pydantic-settings`, include it in dependencies; if using only `pydantic`, avoid untyped global env parsing scattered through modules.

### Testing Requirements

- Add a focused `pytest` smoke test using FastAPI `TestClient` or equivalent.
- Test must cover:
  - `GET /api/v1/health` returns `200`.
  - response includes `status == "ok"`.
  - `/openapi.json` returns `200` and contains the health path.
- Keep test deterministic and independent of PostgreSQL, Qdrant, Telegram, LLM, OCR or external network.
- Do not require Docker to run the health smoke test.

### Documentation Requirements

README update should include:

- Python 3.13 requirement.

- Local API run command, e.g. `uvicorn app.main:app --reload` or project script equivalent.

- Health URL and docs URL:

### Latest Technical Notes

- FastAPI official release notes in February 2026 dropped Python 3.9 support and current FastAPI packages require Python >=3.10, so Python 3.13 is compatible with the project direction; avoid old snippets that assume Python 3.8/3.9-era FastAPI examples. Source: https://fastapi.tiangolo.com/release-notes/
- FastAPI PyPI currently advertises `Requires: Python >=3.10`; do not pin to stale versions solely to preserve old Python support. Source: https://pypi.org/project/fastapi/
- aiogram 3 documentation describes it as async and Python 3.10+; do not use aiogram 2.x handler patterns in future bot scaffolding. Source: https://docs.aiogram.dev/
- pytest 9 documentation/release notes are available; use normal pytest layout and avoid deprecated config patterns if adding pytest config. Source: https://docs.pytest.org/en/9.0.x/
- Qdrant local Docker usage is official and later Docker Compose should use the `qdrant/qdrant` image rather than folding vector storage into PostgreSQL. Source: https://qdrant.tech/documentation/installation/
